- **비고**: 이 데이터가 **경쟁강도(동일업종 밀집도) 산출과 지도 시각화(경위도 즉시 사용)에 가장 핵심적인 데이터**로 판단됨. 단, 폐업 여부 컬럼이 없어 "현재 영업 중인 곳"만 담긴 스냅샷으로 추정(별도 확인 필요) — 폐업률은 위 일반음식점표준데이터(2~4번)와 결합해야 산출 가능.

## 12~14. 부산 빅데이터플랫폼 (일별 행정동 시간/업종 생활인구·소비매출 월별 일평균)

> **2026-07-08 업데이트: 1,000행 절단 문제의 원인을 찾아 전체 데이터로 재다운로드 완료.** 아래는 원인 조사 결과와 수정된 최종 데이터 내용.

### 원인 조사 결과

- **원인**: 기존 3개 CSV는 Open API 페이지네이션 문제가 아니라, **부산 빅데이터플랫폼(Big-데이터웨이브, data.busan.go.kr) 웹사이트의 "SHEET(미리보기 그리드)" 화면에서 내보낸 파일**이었음.
  - macOS 파일 메타데이터(`mdls -name kMDItemWhereFroms`)로 확인: 3개 파일 모두 `blob:https://data.busan.go.kr/...` 형태의 Safari 다운로드였음 — blob: URL은 브라우저 내 JS가 화면에 표시된 데이터를 클라이언트에서 파일로 변환해 내려주는 방식으로, 서버가 실제로 내려준 완전한 파일이 아니라 **화면에 로드된 미리보기 그리드(최대 1,000행)를 그대로 내보낸 것**.
  - 이 플랫폼에서 데이터셋은 "파일 데이터(F)" 유형으로 등록되어 있고 **별도의 Open API(서비스키+numOfRows+pageNo)는 제공하지 않음** — 플랫폼 자체 검색 API(`/bdip/srh/getPublicDataListSearch.do`)로 조회한 각 데이터셋의 `DATATY`가 `F S C`(파일·시트·차트)일 뿐 `A`(API)가 없는 것으로 확인.
  - 진짜 "전체 다운로드"는 `POST /bdip/opendata/selectFileData.do {"publicdatapk": "PD_LP00003"}` 호출로 얻는 실제 파일 URL(`downurl`)에서 받아야 하며, 이 URL이 내려주는 파일은 **XLSX 원본**으로 절단이 없음.
- **재현/수정**: `scripts/download_busan_full_data.py` 작성 → `selectFileData.do`로 각 데이터셋의 실제 다운로드 URL 조회 → XLSX 다운로드 → CSV로 변환해 기존 파일을 덮어씀. 절단됐던 원본은 `_incomplete_backup_1000row/`에 보관.

### 수정된 데이터 내용

- **12. 시간 생활인구** (177,360행): 컬럼 `기준년월(202301~202512),행정동코드,행정동명,시간대(00~23시),평균주거인구수,평균직장인구수,평균방문인구수`. **부산 전체 206개 행정동 × 36개월 × 24시간대**. 결측치 0.
- **13. 시간 소비매출** (177,144행): 컬럼 `기준년월,행정동코드,행정동명,시간대,평균이용금액,평균이용건수`. 부산 전체 206개 행정동 × 36개월 × 24시간대(일부 시간대 결측 조합 존재). 결측치 0, 금액 0/음수 없음.
- **14. 업종 소비매출** (81,301행): 컬럼 `기준년월,행정동코드,행정동명,업종대분류,평균이용금액,평균이용건수`(교육/미용/생활/여가문화/여행숙박/유통/음식주점/음식료품/의류잡화/자동차/의료 11개 대분류). 부산 전체 206개 행정동 × 36개월 × 업종. 결측치 0.
- **지역/업종 범위**: **여전히 부산 전용** — 이 플랫폼 자체가 부산광역시(빅데이터통계과) 소유이므로 경남·울산 데이터는 원천적으로 없음 (아래 "경남/울산 확인 결과" 참고).
- **활용 시 주의**: 이제 부산 전체 206개 행정동, 3년치 월별 시계열이 확보되어 PRD의 "유동인구(시간대)", "매출 추정", "최근 폐업률/트렌드" 요구사항에 부산 지역만큼은 충분히 대응 가능. 경남/울산은 여전히 이 레벨의 데이터가 없으므로 인구세대현황(5~8)·상가정보(9~11)·일반음식점표준데이터(2~4)로 대체해야 함.

### 경남/울산 동일 데이터 존재 여부 확인 결과

같은 방식(시간대별·행정동 단위 생활인구/소비매출)의 데이터가 경남·울산에도 있는지 아래 경로로 확인함:

1. **경남빅데이터허브플랫폼(bigdata.gyeongnam.go.kr)**: 부산과 달리 이 플랫폼은 **일부 데이터셋에 실제 Open API(ServiceKey+numOfRows+pageNo)를 제공**하는 것을 확인했음(예: `apiParam ="?ServiceKey=...&numOfRows=5000&pageNo=1"`). 그러나 "행정동별 시간대 생활인구"·"업종별 소비매출" 같은 부산 수준의 통신사/카드사 제휴 데이터셋은 검색에서 발견하지 못했음. "생활인구 패턴분석", "상권분석시스템" 메뉴가 존재하나 대시보드 성격이며 원자료 파일/API 다운로드로는 확인되지 않음.
2. **공공데이터포털(data.go.kr) 전체 검색**: "경상남도 생활인구/유동인구/소비매출" 키워드로 검색한 결과, 도 전체 단위 통합 데이터는 없고 **시/군 단위로 흩어진 개별 데이터셋**만 존재(예: 창원시 월별 구별 유동인구, 양산시 읍면동별 유동인구, 거제시 주민등록인구현황 등) — 부산처럼 전체 시도를 시간대·행정동 단위로 커버하는 단일 데이터셋은 없음.
3. **울산광역시 데이터포털(data.ulsan.go.kr/bigdata)**: 울산연구원이 운영하는 유사 플랫폼이 존재하나, data.go.kr 검색에서 "울산 생활인구/유동인구/소비매출" 관련 항목은 방사능 방재정보 등 무관한 데이터만 매칭되어 부산 수준의 데이터셋을 확인하지 못했음.

**결론**: 부산의 "행정동×시간대 생활인구/소비매출" 데이터는 부산광역시가 통신사·카드사와 개별적으로 맺은 자체 빅데이터 사업(빅데이터통계과)의 산물로 보이며, 경남·울산에는 **동급의 데이터가 공개돼 있지 않음**. 경남·울산은 시/군 단위로 파편화된 유동인구 데이터(창원·양산 등)만 존재.

**대안**:
- 경남·울산은 이번에 확보한 인구세대현황(5~8, 시군구 단위)·상가정보(9~11, 행정동 단위 업종 마스터)·일반음식점표준데이터(2~4, 개폐업 시계열)만으로 배후수요·경쟁강도 점수를 구성하고, "시간대별 유동인구/매출" 항목은 부산에만 표시하거나 경남·울산엔 "데이터 없음" 처리(PRD의 DataProvider fallback으로 Mock/추정치 대체) 권장.
- 시간이 되면 경남빅데이터플랫폼에서 확인된 Open API 패턴(`ServiceKey`+`numOfRows`+`pageNo`)으로 개별 시/군(창원·양산 등) 유동인구 데이터를 추가 수집해 경남 일부 지역만 보강하는 것은 가능.
- SKT/BC카드 등이 별도로 운영하는 전국 단위 유료/제한적 공개 API(지역경제 상권분석서비스 등)를 검토해볼 수 있으나 별도 신청·인증이 필요해 해커톤 시간 내 확보는 어려울 것으로 보임.
- **활용 시 주의**: 이 3개 파일은 시간대별·업종별 실제 소비 패턴(유동인구/매출)을 보여주는 유일한 데이터라 PRD의 "유동인구(시간대)", "매출 추정" 요구사항에 정확히 부합하지만, **부산 일부 지역만 있어 전역 서비스에는 부족** — 데모에서는 이 데이터가 있는 행정동(부산 원도심권)을 우선 시연 지역으로 삼는 것이 안전.

## 종합 인사이트 (스코어링 엔진 설계 참고)

1. **지역 커버리지 격차**: 부산·울산·경남 "전체"를 다루는 데이터는 `일반음식점표준데이터`(2~4)와 `상가(상권)정보`(9~11), `인구세대현황`(5~8) 3종뿐. 유동인구·매출 데이터(12~14)는 재다운로드 후에도 **부산에만 존재**(경남·울산에는 동급 데이터가 공개돼 있지 않음을 확인, 상세 조사 결과는 12~14번 항목 참고)하므로, **PRD 4장의 2-Track 전략(로컬 데이터 우선 → 부족 시 API/목업 fallback)이 실제로 필요한 상황**임. 경남·울산은 배후수요 점수 산출 시 인구세대현황(5~8, 시군구 단위)과 생활인구현황(1, 경남 분기별)으로 대체해야 함.
2. **좌표계 불일치**: 상가(상권)정보(9~11)는 WGS84 경도/위도 그대로 사용 가능하지만, 일반음식점표준데이터(2~4)의 좌표정보(X)/(Y)는 TM 평면좌표로 추정되어 별도 변환이 필요함. 지도 렌더링 시 상가(상권)정보 좌표를 우선 사용하는 것을 권장.
3. **행정동 단위 매칭 필요**: 상가정보(행정동 단위)와 인구세대현황(시군구 단위)의 공간 단위가 달라 조인 시 행정동→시군구 매핑이 필요. 부산 유동인구/매출 데이터(12~14)는 행정동코드가 있어 상가정보(행정동코드)와 직접 조인 가능.
4. **폐업률 산출**: 일반음식점표준데이터의 인허가일자/폐업일자/영업상태명으로 업종별·지역별 폐업률 계산 가능(단, "일반음식점" 업종에 한정 — 카페 등 타 업종의 폐업률은 상가(상권)정보 스냅샷만으로는 계산 불가, 시계열이 아니므로).
5. **재다운로드 완료**: 12~14번 부산 데이터는 1,000행 절단 원인을 찾아 전체 데이터(36개월×206개 행정동)로 교체했음(`scripts/download_busan_full_data.py`). 절단된 원본은 `_incomplete_backup_1000row/`에 보관.

## 서비스 개요 (PRD_여기차려.md 요약)

- **서비스명**: 여기차려 (가칭)
- **한줄 정의**: 사용자가 부울경 지도에서 위치+업종을 선택하면, 상권 빅데이터 기반 AI가 생존 가능성 점수와 이유를 지도 위에 시각화해주는 웹 서비스
- **핵심 플로우**: 지역/업종 선택 → 스코어링 엔진(가중합 또는 경량 ML)이 배후수요·경쟁강도·접근성·수익성 점수 산출 → Claude API가 점수를 근거로 자연어 리포트 생성 → 지도 히트맵 + Top3 입지 카드 + 상세 리포트로 시각화
- **기술 스택(제안)**: React+Kakao Map/Leaflet(프론트), FastAPI(백엔드), SQLite(DB), scikit-learn 또는 가중합(스코어링), Claude API(리포트)
- **데이터 전략**: `DataProvider` 인터페이스로 Local → API → Mock 순 fallback 체인 설계 (이번에 확인한 로컬 데이터의 지역/업종 공백을 이 fallback 구조로 메꾸는 것이 핵심 리스크 대응)
- **성공 기준**: 지역+업종 선택 → 점수 산출 → 지도 시각화 → LLM 리포트까지 엔드투엔드 데모, 점수 로직의 투명성(블랙박스 아님), 실데이터 기반임을 출처와 함께 증명
