# PRD: 여기차려 (임시명)
### 빅데이터·LLM 기반 부울경 예비 소상공인 맞춤형 상권 분석 및 스마트 입지 추천 서비스

- **해커톤**: 부울경 AI+X (AI융합 프로젝트)
- **트랙**: 2. 빅데이터 기반 AI 개발 활용 신수요 창출형 AI융합 프로젝트 / 3. AI기술 활용 문제·이슈 해결 (통합)
- **문서 버전**: v1.0 (해커톤 무박2일용 실행 계획)
- **작성 대상**: Claude Code (개발 에이전트)

---

## 1. 프로젝트 개요

> **📌 데모 범위 조정 (데이터 조사 결과 반영, 2단계 진행 전 확정)**
> 실제 확보 데이터 조사 결과, 부산은 행정동×시간대 단위 유동인구·소비매출 데이터(부산빅데이터플랫폼, 자체 통신사/카드사 제휴 사업)가 존재하나, **경남·울산에는 동급 데이터가 공개되어 있지 않음**을 확인함(시/군 단위로 파편화된 일부 데이터만 존재).
> → **해커톤 데모는 "부산 상권 분석"으로 범위를 확정**하고, 경남/울산은 "데이터 확보 시 즉시 확장 가능한 구조"로 설계했음을 발표에서 어필하는 전략으로 전환. 서비스명/컨셉("부울경")과 DataProvider 확장 구조는 유지하되, 실제 동작 범위만 부산으로 좁힘.

### 1.1 한 줄 정의
사용자가 **부산 지역** 지도에서 **원하는 위치 + 업종**을 선택하면, 상권 빅데이터 기반 AI가 **생존 가능성 점수와 이유를 지도 위에 시각적으로** 보여주는 웹 서비스. (경남/울산은 데이터 확보 시 확장 가능하도록 구조화)

### 1.2 핵심 가치 제안
- 예비 창업자가 "감"이 아니라 **데이터 기반 근거**로 입지를 판단
- 복잡한 공공데이터를 LLM이 **사람 말로 번역**해서 알려줌
- 지도 기반 **직관적 시각화**로 심사위원/사용자 모두 한눈에 이해

### 1.3 1차 목표 (해커톤 데모 범위)
> 사용자가 부울경 내 원하는 구역 + 업종을 선택 → AI가 상권 데이터를 분석 → 격자/행정동 단위로 **생존 예측 점수(잘될 확률/망할 확률)** 를 지도에 색으로 표시 + LLM이 이유를 리포트로 설명

## 2. 사용자 시나리오 (User Flow)

```
1. 사용자가 지도에서 부울경 내 관심 지역을 검색/클릭 (예: "서면")
2. 창업하려는 업종 선택 (예: "커피숍") — 드롭다운 or 자연어 입력
3. [분석하기] 클릭
   ↓
4. 백엔드: 해당 지역의 격자/행정동 단위 데이터 수집·계산
5. 스코어링 엔진: 가중치/ML 모델로 지역별 점수 산출 (0~100점)
6. LLM: 점수 + 원본 데이터를 받아 "왜 그런 점수인지" 자연어 리포트 생성
   ↓
7. 프론트: 지도에 히트맵(색상)으로 점수 표시 + 상위 추천 입지 Top 3 카드
8. 사용자가 특정 격자/지점 클릭 → 상세 리포트(유동인구, 경쟁강도, 임대료 등 + LLM 해설) 팝업
```

### 2.1 예시 대화형 인터랙션 (Stretch Goal)
- 입력창에 "서면에 커피숍 차릴 건데 어디가 좋아?" 자연어 입력 → NLU로 지역/업종 파싱 → 위 플로우와 동일하게 처리

## 3. 핵심 기능 요구사항

### 3.1 [필수] 지역·업종 선택 및 지도 시각화
- 부울경(부산/울산/경남) 범위의 지도 표시 (Kakao Map API 또는 Leaflet + VWorld/공공 지도)
- 행정동 또는 격자(예: 500m×500m) 단위 경계 오버레이
- 업종 선택 UI: 카페, 음식점, 편의점, 미용실 등 주요 업종 프리셋 (드롭다운)
- 사용자가 지도 영역 클릭 or 드래그로 관심 구역 지정

### 3.2 [필수] 상권 데이터 수집·가공
- **1차 우선순위**: 미리 확보한 데이터셋(공공데이터포털/빅데이터플랫폼에서 다운로드한 것) 로컬 DB/파일 기반 사용
- **2차 (데이터 부족 시)**: 실시간 API 연동 (소상공인시장진흥공단 상가업소 API 등)
- 필요 데이터 항목:
  - 유동인구 (시간대/연령/성별)
  - 업종별 사업체 수 및 개폐업 현황 (경쟁 강도, 최근 폐업률)
  - 집객시설 (지하철역, 버스정류장, 학교, 오피스 등)
  - 임대료/매출 추정 데이터 (있으면 가중치에 포함)
  - 배후 인구 (주거인구/직장인구)
- **2-Track 데이터 전략** (아래 4장 참고)

### 3.3 [필수] AI 스코어링 엔진 (핵심 차별점)
- 격자/행정동 단위로 아래 지표를 종합해 **0~100점 생존/성공 스코어** 산출
  - 배후수요 점수 (유동인구, 배후인구)
  - 경쟁강도 점수 (동일업종 밀집도, 최근 폐업률)
  - 접근성 점수 (대중교통, 집객시설 근접도)
  - (확보 데이터 있으면) 매출/임대료 기반 수익성 점수
- **구현 방식 (택1, 시간에 따라 조정)**:
  - **Track A (권장, 시간 충분 시)**: 각 지표를 정규화 → 로지스틱 회귀 또는 XGBoost 등 경량 ML 모델로 "생존/폐업 가능성" 학습 (라벨: 실제 폐업 데이터 있으면 사용, 없으면 가중합 점수를 pseudo-label로 부트스트랩)
  - **Track B (시간 부족 시, fallback)**: 전문가 가중치 기반 가중합 스코어링 (예: 배후수요 0.35 + 경쟁강도 0.3(역가중) + 접근성 0.2 + 수익성 0.15)
  - 두 트랙 모두 **점수 산출 로직을 코드로 명확히 보여줄 수 있어야 함** (심사 대응력)

### 3.4 [필수] LLM 기반 리포트 생성
- 스코어링 엔진의 출력(숫자, 지표별 세부값)을 LLM(Claude API)에 전달
- LLM은 **숫자를 재계산하지 않고 해설만** 담당 → 일관성/신뢰성 확보
- 출력 형식:
  - Top 3 추천 입지 + 각각의 선정 이유 (배후수요, 경쟁강도 등 근거 명시)
  - 리스크 요인 (예: "이 구역은 카페 밀집도가 높아 경쟁이 치열합니다")
  - 사용자 눈높이의 친절한 톤 (전문 용어 최소화)

### 3.5 [필수] 대화형 스마트 리포트 UI
- 지도 + 우측/하단 패널에 리포트 텍스트 표시
- 추천 점수 시각화 (게이지, 색상 뱃지 등)
- Top 3 입지 카드 (클릭 시 지도 해당 위치로 이동 + 상세 데이터 확장)

### 3.6 [Stretch] 자연어 질의 인터페이스
- 자유 텍스트 입력 → LLM이 지역명/업종 파싱 → 3.1~3.5 플로우 자동 실행

### 3.7 [Stretch] 비교 기능
- 여러 후보 지역 나란히 비교 (표 or 레이더 차트)

## 4. 데이터 전략 (2-Track)

> 확보된 데이터 양에 따라 실시간으로 전환 가능하도록 설계 — **API 실패 시 로컬 데이터로 자동 fallback**

| 상황 | 전략 |
|---|---|
| 확보 데이터로 충분 | 로컬 CSV/DB(SQLite) 기반으로 전체 데모 구성 |
| 특정 지역/업종 데이터 누락 | 소상공인시장진흥공단 상가(상권)정보 API 등 실시간 호출로 보완 |
| API도 실패 | 사전에 준비한 대표 지역(예: 서면, 남포동, 해운대 등) 목업 데이터로 자동 전환 |

**권장 조치**: 백엔드에 `DataProvider` 인터페이스를 만들어 `LocalDataProvider` / `ApiDataProvider` / `MockDataProvider`를 동일 인터페이스로 구현 → 상위 로직(스코어링, LLM)은 데이터 소스를 신경 쓰지 않도록 설계 (해커톤 리스크 관리 핵심)

## 5. 시스템 아키텍처

```
[Frontend: React + 지도(Kakao Map/Leaflet)]
        │  (지역, 업종 선택 요청)
        ▼
[Backend API: FastAPI or Express]
        │
        ├─▶ [DataProvider] ── Local DB / 공공데이터 API / Mock (fallback 체인)
        │
        ├─▶ [Scoring Engine] ── 가중합 or 경량 ML 모델 → 격자별 점수
        │
        └─▶ [LLM Report Generator] ── Claude API 호출 (점수+데이터 → 자연어 리포트)
        ▼
[Frontend] 지도 히트맵 렌더링 + 리포트 패널 표시
```

### 5.1 기술 스택 제안
- **Frontend**: React (Vite) + Kakao Map API(국내 지도 정밀도 우수) 또는 Leaflet + VWorld
  - 시각화: 격자 히트맵은 GeoJSON 폴리곤 + 색상 스케일 (d3-scale 활용 가능)
- **Backend**: FastAPI (Python) — 데이터 처리/ML에 유리
- **DB**: SQLite (해커톤 규모에 적합, 설치 부담 없음)
- **AI**: 
  - 스코어링: scikit-learn (LogisticRegression/RandomForest) 또는 순수 가중합 (numpy/pandas)
  - 리포트: Anthropic Claude API (`claude-sonnet-4-6` 등)
- **배포**: 로컬 데모 + (여유 있으면) Vercel/Render 등 간단 배포

## 7. 개발 우선순위 (무박2일 타임라인 제안)

| 시간대 | 작업 |
|---|---|
| D1 오전 | 데이터 정제(확보 데이터 스키마 통일), DataProvider 구조 설계 |
| D1 오후 | 스코어링 로직 구현 (Track B 먼저 완성 → 여유 있으면 Track A 도전) + 지도 기본 렌더링 |
| D1 저녁 | 지도-데이터 연결(히트맵), LLM 리포트 생성 프롬프트 설계 및 연동 |
| D1 심야 | 전체 플로우 통합 (지역·업종 선택 → 점수 → 리포트) |
| D2 오전 | UI/UX 다듬기 (색상, 애니메이션, 반응형), 예외처리(fallback) |
| D2 오전~정오 | 데모 시나리오 리허설, 발표 자료 준비 |

**우선순위 원칙**: 화려한 기능보다 **엔드투엔드로 한 번에 도는 것**이 최우선. 자연어 질의(3.6), 비교기능(3.7)은 시간 남으면.

## 8. 리스크 및 대응

| 리스크 | 대응 |
|---|---|
| 확보 데이터에 특정 지역/업종 없음 | DataProvider fallback 체인으로 API/목업 자동 전환 |
| LLM API 응답 지연/실패 | 타임아웃 설정 + 실패 시 스코어만 표시하는 기본 리포트 템플릿 fallback |
| 격자 단위 지도 시각화 구현 시간 부족 | 행정동 단위로 단순화 (GeoJSON 행정동 경계 공공데이터 활용) |
| ML 모델 학습 데이터(라벨) 부족 | Track B(가중합)로 시작 → 시간 남으면 가중합 점수를 pseudo-label로 삼아 Track A 실험 |

## 9. Claude Code 작업 지시 요약

Claude Code로 개발 시 아래 순서로 요청하는 것을 권장합니다:

1. 프로젝트 스캐폴딩 (FastAPI 백엔드 + React 프론트 기본 구조)
2. `DataProvider` 인터페이스 + `MockDataProvider` 먼저 구현 (전체 파이프라인 먼저 동작시키기)
3. 지도 렌더링 + GeoJSON 오버레이 (Mock 데이터로 히트맵부터 시각적으로 확인)
4. 스코어링 엔진 구현 (Track B 가중합 우선)
5. Claude API 연동 리포트 생성 (프롬프트: 점수+지표 JSON → 자연어 리포트, 출력 포맷 고정)
6. 실데이터 연결 (`LocalDataProvider`로 교체)
7. UI 폴리싱 및 예외처리

## 10. 성공 기준 (해커톤 심사 대응)

- ✅ 실제로 지역+업종 선택 → 점수 산출 → 지도 시각화 → LLM 리포트까지 **엔드투엔드 데모 가능**
- ✅ 점수 산출 로직을 코드로 설명 가능 (블랙박스 LLM 판단이 아님을 어필)
- ✅ 부울경 실제 데이터 기반임을 보여줄 수 있음 (출처 명시)
- ✅ 지도 시각화의 완성도 (색상, 인터랙션, 반응 속도)
