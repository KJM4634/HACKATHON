# ARCHITECTURE — 여기차려

이 문서는 "여기차려"(부울경 소상공인 입지 추천 서비스)의 코드 구조를 팀원이
빠르게 파악할 수 있도록 정리한 것입니다. 데이터/PRD 배경은
[`data_inventory.md`](./data_inventory.md), 데모 시연 순서는
[`README.md`](./README.md)를 참고하세요.

---

## 1. 전체 동작 파이프라인 (5단계)

```
① 사용자 입력          ② 데이터 조회           ③ 점수 계산              ④ 결과 해설 생성        ⑤ 화면 표시
검색 or 자연어 질의  →  DataProvider          →  Track B 가중합         →  Gemini API          →  지도 / 카드 / 리포트
                        (규칙 기반, AI 아님)      + Track A 실험적 ML       (여기가 생성형 AI)
                                                   (규칙·통계 기반, AI 아님)
```

| 단계 | 무엇을 하는가 | AI가 관여하는가 |
|---|---|---|
| ① 사용자 입력 | 검색창은 문자열 그대로 매칭(결정론적). **자연어 질의**("서면에 카페 차릴 건데 어디가 좋아?")는 지역/업종을 추출해야 해서 Gemini를 씀 | **예 — AI 지점 #1** (`app/llm/query_parser.py`) |
| ② 데이터 조회 | 행정동 코드로 인구/유동인구/경쟁업체/폐업률 CSV를 찾아 읽어옴 | 아니오 — 그냥 파일 읽기/필터링 |
| ③ 점수 계산 | 정해진 수식(가중합)과 학습된 통계 모델(RandomForest)로 점수 산출 | 아니오 — 수식·통계 계산 (모델은 통계 기반이지 대화형 AI가 아님) |
| ④ 결과 해설 생성 | 계산된 점수를 자연어 문장으로 "설명"만 함 | **예 — AI 지점 #2** (`app/llm/report.py`, `app/llm/strategy.py`) |
| ⑤ 화면 표시 | React가 위 결과를 지도/카드/패널로 그림 | 아니오 |

**핵심**: 이 서비스의 점수(총점/배후수요/경쟁강도/수익성)는 **AI가 지어내는 게 아닙니다.** ②③은 100% 규칙·통계 기반이라 같은 지역·업종이면 언제 조회해도 같은 점수가 나옵니다(재현 가능·검증 가능). Gemini는 자연어 질의 파싱과 결과 해설(④)에 쓰이고, ④에서는 "주어진 숫자를 다시 계산하지 말고 그대로 인용해서 해설만 하라"고 프롬프트에서 명시적으로 금지해뒀습니다.

격자 셀 상세 패널에서 "이 지역 리뷰 요약 보기"를 누르면 세 번째 AI 관여 지점이 추가로 동작합니다 — **AI 지점 #3**(`app/llm/review_analyzer.py`): 네이버 블로그 검색 API로 그 행정동의 블로그 리뷰를 가져와 Gemini가 요약합니다. ①②의 두 지점과 달리 점수 계산과는 완전히 무관한 참고 정보라, 실패해도 다른 정보(게이지·막대바·AI 해설)에는 영향을 주지 않습니다.

## 2. Backend 구조 (`backend/app/`)

### 진입점 · 공용

| 경로 | 역할 | 왜 필요한가 |
|---|---|---|
| `main.py` | FastAPI 앱 생성, 라우터 등록, CORS 설정 | 서버가 실행될 때 제일 먼저 도는 파일 |
| `schemas.py` | 모든 API 요청/응답 데이터 모델(Pydantic) | 프론트-백엔드가 주고받는 JSON 모양을 한 곳에서 관리 — 필드 하나 바뀌면 여기부터 봄 |

### 데이터 조회 (② 단계, `data_provider/`)

| 경로 | 역할 | 왜 필요한가 |
|---|---|---|
| `data_provider/base.py` | `DataProvider` 추상 인터페이스 | 점수 계산 코드가 "데이터가 CSV에서 오는지 API에서 오는지"를 몰라도 되게 하는 경계선. 나중에 데이터 소스가 바뀌어도 이 인터페이스만 지키면 나머지는 안 건드려도 됨 |
| `data_provider/mock_provider.py` | 서면/남포동/해운대/광안리 4곳 고정 가짜 데이터 | 초기 개발 단계에 실제 CSV 없이도 화면을 만들 수 있었던 이유. 지금은 실제 호출 경로에서 안 쓰이고 비상용 폴백으로만 남아있음 |
| `data_provider/local/local_provider.py` | **지금 서비스가 실제로 쓰는 구현체** — 부산 206개 행정동 전체를 실제 CSV로 처리 | 여기서 경쟁업체수/폐업률/인구/유동인구를 실제로 계산해서 리턴함 |
| `data_provider/local/category_mapping.py` | 업종(카페/한식/중식/분식/기타음식점/편의점/미용실) → 원본 CSV의 실제 분류값 매핑 | "음식점"은 UI에서만 존재하는 묶음이고, 실제로는 한식/중식/분식/기타음식점 4개로 세분화돼 있음(일식·양식은 표본이 얇아 기타음식점에 흡수) — 이 매핑이 그 기준점 |
| `data_provider/local/sanggabu_loader.py` | 소상공인시장진흥공단 상가(상권)정보 CSV 로더 | 경쟁업체수 계산의 원천(좌표 있음) |
| `data_provider/local/restaurant_loader.py` | 일반음식점표준데이터 CSV 로더 + TM좌표 변환 | 폐업률 계산의 원천 |
| `data_provider/local/population_loader.py` | 구 단위 인구세대현황 CSV 로더 | 배후수요(인구) 계산의 원천. 행정동 단위 데이터가 없어서 "구 단위 추정치"라는 한계가 여기서 생김 |
| `data_provider/local/foot_traffic_loader.py` | 부산 시간대별 생활인구(유동인구) CSV 로더 | 배후수요(유동인구) 계산의 원천 |
| `data_provider/local/consumption_loader.py` | 부산 시간대·업종별 소비매출 CSV 로더 | 수익성 계산의 원천 |
| `data_provider/local/dong_mapper.py` | 법정동 주소 텍스트 → 행정동 코드 매핑(KDTree 최근접) | 일반음식점표준데이터에는 행정동 필드가 없고 주소 텍스트만 있어서, 애매한 경우(법정동 하나가 행정동 여러 개로 쪼개진 경우) 좌표로 최근접 매칭함 |
| `data_provider/local/dong_boundary_loader.py` | 지도용 GeoJSON(행정동 경계) 로더 — 면적 계산, bbox, "이 점이 이 동 안에 있는가" 판정 | 격자 확대 모드(`grid.py`)가 "이 동을 몇 m 격자로 나눌지"와 "격자가 실제 동 모양 안에 있는지"를 판단하는 데 씀 |
| `data_provider/geo.py` | 좌표계 변환(TM5174 → WGS84) | 일반음식점표준데이터의 좌표가 지도 표준(WGS84)이 아니라서 변환이 꼭 필요함 |

### 점수 계산 (③ 단계, AI 아님)

| 경로 | 역할 | 왜 필요한가 |
|---|---|---|
| `scoring.py` | **Track B**: 배후수요·경쟁강도·수익성 가중합 (접근성은 데이터가 없어 제외, 나머지 비율대로 재분배) | 이 서비스 점수의 본체. 정규화 상/하한은 부산 206개 행정동 실측 분포(P5~P95)로 잡음 |
| `ml/predict.py` | **Track A**: RandomForest로 학습한 "한식 3년 내 폐업 확률" 실험적 예측 | 한식만 실제 라벨(인허가일자/폐업일자)이 있어 이 업종에만 적용. 메인 점수와는 독립적으로 참고용으로만 노출 |
| `alternatives.py` | 점수가 낮은(50점 이하) 지역에 대해 3km 이내·더 높은 점수의 대안 지역 찾기 | "여기 말고 저기가 낫다"를 지도에서 바로 보여주기 위한 로직 |
| `region_aliases.py` | "서면"→부전1/2동+전포1/2동 같은 생활권 이름 별칭표 | 공식 행정동명이 아닌 생활권 이름을 검색창/자연어 질의 양쪽에서 항상 같은 결과로 매칭시키기 위함 (Gemini를 매번 부르면 결과가 들쭉날쭉했음) |
| `grid.py` | 행정동 하나를 클릭했을 때 그 동만 100~1000m 격자로 잘라 점수 재계산 | 행정동보다 더 촘촘한 "골목 상권" 단위 비교. 인구/유동인구는 행정동 단위뿐이라 그 동의 상가업소 분포로 근사 배분(2단계 근사) |
| `budget.py` | 사용자가 입력한 예상 월세와 수익성 점수를 교차해 "여유/보통/부담" 참고 문구 생성 | `total_score` 계산에는 전혀 반영되지 않는 보조 배지(`BudgetFit`). 개별 점포 단위 임대료 공공데이터가 없어 확정적 계산 대신 상대적 판단만 함 |
| `trend.py` | 행정동 단위 "최근 추세" 배지 — 이 동의 최근 매출 YoY 증감률을 부산 전체 중앙값과 비교 | 이 역시 `total_score`와 무관한 보조 정보(`TrendFit`, budget.py와 같은 패턴). 절대 증감률만 보면 "다들 오르는 시기라 오른 것"과 구분이 안 돼 반드시 상대값으로 판단 |

### 결과 해설 생성 (④ 단계, 여기가 AI)

| 경로 | 역할 | 왜 필요한가 |
|---|---|---|
| `llm/query_parser.py` | 자연어 문장에서 지역/업종 추출 (**AI 지점 #1**) | "서면에 카페 차릴 건데 어디가 좋아?" 같은 문장을 검색창 입력값(지역+업종)으로 변환 |
| `llm/report.py` | 점수+세부지표를 근거로 Top3 추천 리포트 작성 (**AI 지점 #2**) | 숫자를 사람이 읽을 문장으로 "해설"만 함 — 점수를 다시 계산하지 말라고 프롬프트에서 금지. Gemini 실패 시 점수만으로 만든 기본 템플릿으로 대체 |
| `llm/strategy.py` | 점수가 낮은 지역의 "차별화 전략" 제안 (**AI 지점 #2**) | report.py와 달리 실패해도 대체 문구를 지어내지 않고 그냥 비워둠(창작 조언은 억지로 템플릿화할 근거가 없어서) |
| `llm/review_analyzer.py` | 네이버 블로그 검색 API로 그 행정동의 방문자 리뷰를 가져와 Gemini로 요약 (**AI 지점 #3**) | 격자 셀 상세의 "이 지역 리뷰 요약 보기"를 눌렀을 때만 호출됨 — 점수 계산과 무관한 참고 정보라 실패해도 다른 정보에 영향 없음 |

### API 라우터

| 경로 | 역할 |
|---|---|
| `api/analyze.py` | 핵심 엔드포인트 5개(`/regions`, `/scores`, `/analyze`, `/report`, `/parse-query`) — 아래 4번 표 참고 |
| `api/grid.py` | 격자 확대 모드 엔드포인트 2개(`/grid`, `/grid/cell`) |

## 3. Frontend 구조 (`frontend/src/`)

| 경로 | 역할 | 왜 필요한가 |
|---|---|---|
| `App.jsx` | 전체 상태 관리(선택된 업종/지역, 분석 결과, 모달, 격자 모드) + 레이아웃 | 이 앱의 "뇌" — 나머지 컴포넌트는 대부분 App.jsx가 내려주는 데이터를 그리기만 함 |

| `colorScale.js` | 점수(0~100) → 빨강~초록 색상 변환 | 지도 히트맵과 게이지 색이 항상 같은 규칙을 쓰게 함(적록색맹 고려해 중간에 주황/노랑을 넣은 4단계 그라디언트) |

| `components/DongMap.jsx` | Leaflet 지도 — 206개 행정동 히트맵, 대안 연결선, **격자 오버레이**(행정동 클릭 시) | 이 서비스의 핵심 시각화. 격자 모드일 때는 나머지 행정동을 흐리게 처리하고 클릭한 동만 사각형 격자로 확대해서 보여줌 |
| `components/AnalysisPanel.jsx` | TOP3 추천 카드 + AI 분석 리포트 텍스트 | "분석하기"를 눌렀을 때 사이드 패널에 뜨는 기본 화면 |
| `components/RegionDetailModal.jsx` | 행정동 하나의 상세 정보 — 게이지, 세부지표 막대바, 대안 추천 탭, "이곳에서 성공하려면"(차별화 전략) 탭, Track A 실험적 모델 섹션 | 지도나 카드를 클릭했을 때 사이드 패널에 뜨는 상세 화면 |
| `components/GridCellDetailPanel.jsx` | 격자 셀 하나의 상세 정보 — 게이지, 막대바, 같은 행정동 내 대안 격자, "AI 해설 보기", "이 지역 리뷰 요약 보기"(네이버 블로그 + Gemini). 패널 제목은 좌표를 그대로 노출하지 않고 "OO동 내 선택 구역"으로 표기 | RegionDetailModal과 같은 자리에 뜨지만, Track A는 일부러 안 넣음(행정동 단위로 학습된 모델이라 격자엔 안 맞음). AI 해설/리뷰 요약 둘 다 버튼을 눌렀을 때만 호출(무료 쿼터 절약). 선택한 셀은 지도(DongMap.jsx)에서 파란 테두리로 표시(격자 목록용 주황색 하이라이트와 구분) |
| `components/QueryBar.jsx` | 자연어 질의 입력창 | "OO에 카페 차릴 건데 어디가 좋아?" 같은 문장을 받아 백엔드 `/api/parse-query`로 보냄 |
| `useCountUp.js` | 게이지 점수를 0에서 목표값까지 부드럽게 증가시키는 카운트업 훅 | 점수가 뜨자마자 뚝 나타나는 대신 애니메이션으로 강조하기 위함 |
| `useDelayedFlag.js` | 특정 상태(로딩 등)가 N초 이상 지속되면 true가 되는 훅 | AI 해설/리뷰 요약처럼 외부 API 호출이 길어질 때 "멈춘 게 아니다"라는 안내 문구로 바꾸는 데 씀(AnalysisPanel/RegionDetailModal/GridCellDetailPanel 공통) |

## 4. 주요 API 엔드포인트

| 메서드 | 경로 | 무엇을 하는가 |
|---|---|---|
| GET | `/api/regions` | 부산 206개 행정동의 기본 정보(이름/좌표) 목록 — 검색창 자동완성/매칭에 씀 |

## 5. 데이터 흐름 상세 (사용자 요청 → 최종 응답)

### 흐름 A — 자연어 질의로 특정 지역을 물어볼 때

```
1. 사용자: "서면에 카페 차릴 건데 어디가 좋아?" 입력 → [질문하기]
2. 프론트(QueryBar.jsx) → POST /api/parse-query
3. 백엔드(query_parser.py): "서면"이 REGION_ALIASES에 있는 단어 하나뿐이면
   Gemini 호출 없이 바로 매칭. 문장 전체를 봐야 하면 Gemini 호출 →
   지역 후보 목록 + 업종을 추출, /api/regions 목록과 대조해 검증
4. 응답: 지역이 정확히 1곳으로 좁혀지면 → 프론트가 그 지역의 상세 화면으로 바로 이동
5. 프론트(App.jsx openRegionDetail) → POST /api/report (지역 1곳) 와
   GET /api/grid (같은 지역) 를 동시에 호출
6. 백엔드(analyze.py): LocalDataProvider로 원자료 조회(②) → scoring.py로
   점수 계산(③) → report.py가 Gemini로 리포트 문장 생성(④)
   백엔드(grid.py): 같은 지역을 격자로 쪼개 각 셀 점수 계산(③과 동일한
   원리, 정규화만 "이 동 안 격자들끼리" 상대적으로 함)
7. 프론트: 리포트는 사이드 패널에, 격자는 지도에 동시에 표시(⑤)
```

### 흐름 B — 업종만 고르고 "분석하기"를 눌렀을 때 (지역 특정 안 함)

```
1. 사용자: 업종 탭(예: 한식) 선택 → [분석하기]
2. 프론트 → GET /api/scores?category=한식 (206개 행정동 전체 가벼운 점수)
3. 프론트: 점수 높은 순 TOP3 지역 ID를 뽑음
4. 프론트 → POST /api/report (TOP3 지역들) → Gemini가 Top3 비교 리포트 작성
5. 프론트: TOP3 카드 목록 + 리포트 텍스트를 사이드 패널에 표시,
   지도는 이미 2번 응답으로 전체 히트맵이 칠해져 있음
```

## 8. 데이터 한계 및 정직하게 다룬 부분 (요약)

자세한 항목별 설명은 서비스 화면 하단 **"데이터 안내"** 접이식 섹션(`App.jsx`의 `DataNoticeFooter`)과 [`README.md`](./README.md)를 참고하세요. 여기서는 코드 구조상 어디서 이런 한계가 생기는지만 짧게 요약합니다.

- **부산만 지원**: 경남·울산은 부산만큼 세밀한 유동인구/소비매출 데이터가 공개돼 있지 않음(`data_inventory.md` 참고) — 지금은 부산 206개 행정동만 다룸.
- **배후인구는 구 단위 추정치**: `population_loader.py`가 행정동보다 큰 시/군/구 단위 CSV밖에 없어서, 같은 구에 속한 모든 행정동에 같은 인구값을 그대로 적용(`PopulationStats.is_gu_level_estimate=True`).
- **Track A는 실험적**: `ml/predict.py`는 한식만 실제 폐업 라벨이 있어 그 업종에만 적용되고, 메인 점수(Track B)와는 독립적으로 참고용으로만 노출됨.
- **"음식점" 세분화**: 원래 "음식점" 하나였던 카테고리를 한식/중식/분식/기타음식점 4개로 나눴는데, 일식·양식은 행정동별 표본이 너무 얇아(조사 결과 절반 가까운 동에서 5건 미만) 기타음식점에 흡수시킴(`category_mapping.py`).
- **격자는 2단계 근사**: `grid.py`의 격자 점수는 구 단위 인구 추정치 위에, 그 동의 상가업소 분포로 인구/유동인구를 한 번 더 근사 배분한 것이라, 부산 전체 기준이 아니라 **같은 행정동 안에서의 상대 비교**로만 의미가 있음.

## 9. 가중치 조정의 통계적 한계

`scoring.py`의 배후수요:경쟁강도:수익성 가중치는 한 번에 정해진 게 아니라 아래 흐름을 거쳐 지금 값(0.3925 : 0.420 : 0.1875)에 도달했습니다. 코드 안 주석(`scoring.py` 모듈 docstring)에도 같은 히스토리가 남아있습니다.

### 조정 과정

1. **PRD 원안**: 배후수요 0.35 : 경쟁강도 0.30 : 접근성 0.20 : 수익성 0.15. `PRD_여기차려.md` 3.3절을 다시 확인해보니 "전문가 가중치 기반 (**예**: ...)"라고만 적혀 있을 뿐, 실제 설문·연구·업계 자료 등 별도 근거는 없는 예시값이었습니다.
2. **접근성 제외 후 재분배**: 접근성 데이터(대중교통·집객시설 근접도)를 확보할 수 없어 제외하고, 나머지 비율(0.35:0.30:0.15)을 유지한 채 배후수요 0.4375 / 경쟁강도 0.375 / 수익성 0.1875로 재분배.
3. **Track A 신호 반영 검토**: 이미 학습된 Track A(RandomForest, 한식 폐업위험 예측) 모델의 feature importance를 뽑아보니, 경쟁강도(특히 폐업률)가 배후수요보다 훨씬 중요하다는 **반대 방향** 신호가 나왔습니다(정규화하면 경쟁강도 68.7% : 배후수요 31.2%). 수익성은 Track A에 해당 변수 자체가 없어 비교 대상이 아니었습니다. PRD 원안 비율(53.8:46.2)과 이 신호(31.2:68.8)를 50:50으로 블렌드해 경쟁강도 0.4669까지 올리는 안을 검토했습니다.
4. **상관관계 분석으로 구조적 한계 확인**: 206개 행정동 전체로 재계산해보니, 경쟁강도 비중을 올릴수록 "배후수요·수익성이 낮은데 경쟁강도만 높게 나오는" 지역(부산진구 개금2동 등)이 부당하게 상위권으로 뛰는 부작용이 나타났습니다. 왜 그런지 통계적으로 확인하기 위해 배후수요·경쟁강도·수익성 세 지표의 상관계수를 계산했습니다.
5. **최종 조정**: 안전장치(배후수요/수익성 하한 게이팅으로 경쟁강도의 긍정적 효과를 제한하는 구조적 로직)까지는 도입하지 않고, 0.42~0.43 범위 안에서 부작용이 가장 작은 지점을 채택하기로 결정했습니다. 이 범위 안에서 경쟁강도 비중을 0.420/0.423/0.425/0.428/0.430로 각각 계산해보니 부작용(배후수요·수익성 게이트 이하 지역 중 점수가 오히려 오르는 지역의 개수·상승폭)이 단조증가했기 때문에, 범위 하한인 **0.420**을 최종 채택했습니다.

### 상관관계 분석 결과

Pearson 상관계수(206개 행정동, 한식 기준 — 원본 데이터/재현 스크립트는 `analysis/breakdown_206_한식.csv`, `analysis/correlation_matrix.csv`):

| | 배후수요 | 경쟁강도 | 수익성 |
|---|---|---|---|
| **배후수요** | 1.000 | **-0.521** | 0.646 |
| **경쟁강도** | -0.521 | 1.000 | -0.495 |
| **수익성** | 0.646 | -0.495 | 1.000 |

- **배후수요 ↔ 경쟁강도: -0.521**(중간 강도 음의 상관) — 유동인구·배후인구가 많은 동네일수록 경쟁도 치열해지는(경쟁강도 점수가 낮아지는) 경향이 뚜렷합니다. "사람이 많이 모이는 곳 = 경쟁업체도 몰리는 곳"이라는 상식과 일치합니다.
- **배후수요 ↔ 수익성: 0.646**(강한 양의 상관) — 유동인구가 많은 곳일수록 매출도 높은 경향.
- **경쟁강도 ↔ 수익성: -0.495**(중간 강도 음의 상관) — 위 두 관계의 결과.

배후수요·수익성이 서로 강하게 붙어 움직이고, 이 둘은 경쟁강도와 반대로 움직입니다. 하지만 이 관계는 "중간 강도"일 뿐 1.0이 아니라서, **상관관계를 벗어나는 예외 지역이 실제로 존재**합니다. 회귀선 잔차가 큰 지점(산점도: `analysis/scatter_demand_vs_competition.png`)을 뽑아보면 다음 6곳이 나옵니다 — 이는 개금2동 하나만의 우연한 예외가 아니라 **구조적으로 반복되는 패턴**이라는 뜻입니다.
