# 여기차려

부산(현재는 부산만 지원) 소상공인을 위한 입지 추천 서비스. 업종을 고르면 206개
행정동 전체를 점수화해 지도에 색으로 보여주고, 후보 지역 TOP3에 대해 Gemini가
자연어 리포트(추천 이유 + 리스크)를 써준다.

## 실행 방법

```bash
# 백엔드 (포트 8000)
cd backend
uv run uvicorn app.main:app --port 8000
# backend/.env 에 GEMINI_API_KEY=... 필요 (없으면 자동으로 점수 기반 기본 리포트로 대체됨)

# 프론트엔드 (포트 5173)
cd frontend
npm run dev
```

http://localhost:5173 접속.

## 30초 데모 시연 순서

1. 처음 화면: 부산 206개 행정동이 카페 기준 점수로 이미 색칠되어 있음 (빨강=위험, 초록=양호)
2. 검색창에 **"해운대"** 입력 → **[분석하기]** 클릭
   → 지도에서 해운대구 소속 동 3곳이 파란 테두리로 강조되고, 우측에 TOP3 카드 + AI 리포트가 뜬다
3. TOP3 카드 1번(해운대구 우3동, 74점) 클릭 → 상세 팝업에서 점수 게이지 + 세부지표 바차트 + AI 해설 확인
4. 업종을 **"음식점"**으로 바꾸고 다시 **[분석하기]** → 같은 해운대구 후보들의 순위가 바뀌는 걸 보여주며 "업종마다 좋은 자리가 다르다"는 포인트 전달
5. 방금 연 상세 팝업 하단의 **"실험적 예측 모델"**을 펼쳐 Track A(실제 폐업 데이터로 학습한 ML 모델) 결과를 보여주고, 신뢰도가 낮다고 명시된 부분을 짚어준다 → 6번의 "정직함" 얘기로 자연스럽게 연결

## 추천 시연 조합

검색 없이 "분석하기"를 누르면 부산 전체 206곳 중 TOP3를 고르는데, 상위권이 대부분
이름이 낯선 동(화명1동, 온천3동 등)이라 데모에서 임팩트가 약하다. **검색창에 구/동
이름을 입력해 후보를 좁히면** 관객이 아는 지명 안에서 비교가 되어 훨씬 설득력 있다.

| 검색어 | 업종 | TOP3 결과 | 비고 |
|---|---|---|---|
| **해운대** | **카페** | 우3동 74 / 좌4동 74 / 반여4동 71 | 배후수요·수익성 모두 높음, 1순위 추천 |
| **해운대** | **음식점** | 우2동 77 / 재송1동 77 / 우1동 75 | Track A(실험적 예측 모델) 섹션도 같이 보여줄 수 있음 |
| (검색 없음) | 카페 | 화명1동 82 / 온천3동 78 / 부암3동 75 | "AI가 부산 전체에서 찾은 의외의 곳"으로 쓸 수도 있음 |

참고로 이 프로젝트에서 계속 기준으로 썼던 4곳(서면/남포동/해운대/광안리)을 카페로
비교하면 서면 부전2동 50점, 남포동 37점, 해운대 우1동 57점, 광안리 광안2동 34점으로
남포동·광안리가 낮게 나온다 — 아래 데이터 한계 설명과 바로 이어붙이기 좋은 소재다.

## 데이터 한계를 정직하게 다룬 부분

이 프로젝트는 확보한 데이터가 실제로 무엇을 말해줄 수 있고 없는지를 숫자와 UI에
그대로 드러냈다. 데모에서 이 부분을 짚어주면 "그럴듯하게 포장하지 않았다"는 신뢰를
준다.

- **범위를 부산으로만 명확히 좁혔다.** 데이터 폴더에는 울산·경남 원본 파일도 있지만,
  법정동→행정동 매핑(좌표 기반 KDTree 매칭)을 검증까지 마친 건 부산뿐이라 그 이상은
  다루지 않았다. "일단 다 되는 것처럼 보이게" 하지 않고 검증된 범위만 서비스한 것.
- **남포동처럼 구 인구가 작은 동은 배후수요 추정이 특히 거칠다.** 인구 데이터가
  행정동이 아니라 구 단위로만 존재해서 모든 동에 소속 구 인구를 그대로 적용하는데,
  남포동이 속한 중구는 부산에서 인구가 가장 적은 구(3.6만 명, 다른 구의 1/10 수준)라
  상권은 활발해도 "배후인구"로는 낮게 잡힌다. 지도/팝업에 점선 테두리로 "구 단위
  추정치"라고 항상 표시한다.
- **접근성 지표는 데이터가 아예 없어 점수에서 뺐다.** 대중교통 정류장·집객시설
  근접도를 계산할 원본 데이터가 없다는 걸 확인하고, 억지로 대체 지표를 끼워 넣는
  대신 가중치를 나머지 지표에 재분배하고 UI에 "데이터 없음" 해치 패턴으로 표시했다.
- **Track A(ML 모델)는 실제 라벨로 학습했지만 성능이 약하다는 걸 그대로 보여준다.**
  일반음식점표준데이터의 실제 인허가일자/폐업일자로 "3년 내 폐업" 라벨을 만들어
  랜덤포레스트를 학습(AUC 0.656)했는데, 행정동 단위 집계 피처만으로는 개별 매장의
  운명을 정확히 맞추기 어렵고 Track B 가중합 점수와 순위 방향도 잘 안 맞는다
  (206개 행정동 전체 Spearman 상관 0.158, 약함). 이걸 숨기지 않고 상세 팝업의
  별도 접이식 "실험적 예측 모델" 섹션에 "신뢰도가 낮다"는 문구와 함께 분리해서
  보여준다 — 메인 점수·지도 색상에는 영향을 주지 않는다. 음식점(한식) 외 업종은
  애초에 폐업 이력 데이터가 없어 이 모델을 적용하지 않는다는 것도 명시한다.
- **LLM은 숫자를 다시 계산하지 않고 해설만 한다.** 프롬프트에서 점수 재계산을
  명시적으로 금지해, 리포트 문장과 실제 점수 로직이 항상 일치하게 만들었다.

### 예외 상황이 실제로 매끄럽게 처리되는지 점검한 것

- Gemini API 키를 일부러 무효화해서 테스트 → 실패를 감지해 점수 기반 기본 리포트로
  자동 전환되고, 화면에는 "AI 리포트 생성에 실패해 기본 요약을 표시합니다" 안내만
  뜬다(에러 화면이나 멈춤 없음)
- 백엔드를 완전히 내려서 테스트 → "서버에 연결할 수 없습니다" 같은 이해할 수 있는
  한국어 메시지로 안내(브라우저 원본 에러 문구 노출 안 함)
- 검색어가 아무 지역과도 안 맞을 때 → "~와 일치하는 지역이 없습니다" 안내
- 데이터가 없는 업종(카페/편의점/미용실)을 골라도 죽지 않고, 접근성처럼 "데이터
  없음"으로 투명하게 표시하며 나머지 점수로 계속 동작
- 데스크톱/태블릿/모바일 너비에서 콘솔 에러 없이 레이아웃이 깨지지 않는지 확인
  (모바일에서 상단바가 겹치던 문제와 지도가 눌리던 문제를 발견해 이번에 수정)
