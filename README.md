# 여기차려

부산(현재는 부산만 지원) 소상공인을 위한 입지 추천 서비스. 업종을 고르면 206개
행정동 전체를 점수화해 지도에 색으로 보여주고, 후보 지역 TOP3에 대해 Gemini가
자연어 리포트(추천 이유 + 리스크)를 써준다. 자연어로 바로 질문할 수도 있고
("서면에 커피숍 차릴 건데 어디가 좋아?"), 선택한 지역 점수가 낮으면 AI가 인근
대안을 찾아 지도에서 직접 연결해 보여준다.

## 실행 방법

```bash
# 백엔드 (포트 8000)
cd backend
uv run uvicorn app.main:app --port 8000
# backend/.env 에 GEMINI_API_KEY=... 필요 (없으면 자동으로 점수 기반 기본 리포트로 대체됨)

# 프론트엔드 (포트 5173)
cd frontend
npm run dev
```

http://localhost:5173 접속.

## 30초 데모 시연 순서

새로 추가된 자연어 질의 + AI 대안 추천 + 지도 연결선을 한 흐름에서 보여주는
버전. **남포동 + 카페**(37점, 자동으로 대안 3곳이 걸리는 조합)를 그대로 쓰면
아래 순서가 실제로 이렇게 동작한다(직접 검증 완료).

1. 처음 화면: 부산 206개 행정동이 카페 기준 점수로 이미 색칠되어 있음 (빨강=위험, 초록=양호)
2. 자연어 입력창에 **"남포동에 카페 하나 내려는데 어때?"** 입력 → **[질문하기]**
   → Gemini가 지역(중구 남포동)과 업종(카페)을 정확히 추출해 자동으로 채우고
   바로 분석까지 실행한다(별도 클릭 없음). TOP1 카드에 **"⚠ 대안 3곳"** 배지가 뜬다
3. TOP1 카드 클릭 → 상세 팝업: 종합 점수 37점 게이지, 세부지표 바차트, 그리고
   **"이 지역보다 나은 인근 대안"** 섹션에 부평동(39점·0.52km) 등 3곳이 카드로 나열됨.
   AI 해설이 "이런 이유로 남포동보다는 [지역]을 추천드립니다"로 결론부터 말하고,
   대안마다 구체적 근거(경쟁강도·수익성 수치)를 한 문장씩 붙이는 것을 짚어준다.
   팝업 뒤로 지도가 이미 원본(빨강)+대안(초록) 위치가 다 보이게 자동 확대돼 있다
4. 팝업 닫기 → 지도에 빨강-초록 점선 연결선이 그대로 남아있음(은은하게 흐르는
   애니메이션). 초록 마커에 마우스 올리면 점수·거리 툴팁, 클릭하면 그 지역 상세로 이동
5. (시간 남으면) 검색창에 **"해운대"** 입력 후 업종을 **"음식점"**으로 바꿔 **[분석하기]**
   → 우2동 77점 카드를 열어 하단의 **"실험적 예측 모델"**을 펼치고, Track A(실제
   폐업 데이터로 학습한 ML)의 신뢰도가 낮다고 명시된 부분을 짚어 "정직함" 얘기로 연결

## 추천 시연 조합

검색/질의 없이 "분석하기"만 누르면 부산 전체 206곳 중 TOP3를 고르는데, 상위권이
대부분 이름이 낯선 동(화명1동, 온천3동 등)이라 데모에서 임팩트가 약하다. **자연어
질의나 검색창으로 후보를 좁히면** 관객이 아는 지명 안에서 비교가 되어 훨씬 설득력 있다.

| 입력 방식 | 내용 | 업종 | 결과 | 비고 |
|---|---|---|---|---|
| 자연어 질의 | "남포동에 카페 하나 내려는데 어때?" | 카페 | 남포동 37점, 대안 3곳(부평동 39·남부민1동 42·초장동 42) | 자연어+대안추천+지도연결선을 한 번에 보여주는 메인 데모 |
| 검색창 | 해운대 | 카페 | 우3동 74 / 좌4동 74 / 반여4동 71 | 전부 고득점 → 대안 추천 미발동(기존 방식) 대비용 |
| 검색창 | 해운대 | 음식점 | 우2동 77 / 재송1동 77 / 우1동 75 | Track A(실험적 예측 모델) 섹션도 같이 보여줄 수 있음 |
| (입력 없음) | - | 카페 | 화명1동 82 / 온천3동 78 / 부암3동 75 | "AI가 부산 전체에서 찾은 의외의 곳"으로 쓸 수도 있음 |

참고로 이 프로젝트에서 계속 기준으로 썼던 4곳(서면/남포동/해운대/광안리)을 카페로
비교하면 서면 부전2동 50점, 남포동 37점, 해운대 우1동 57점, 광안리 광안2동 34점으로
남포동·광안리가 낮게 나온다 — 아래 데이터 한계 설명과 바로 이어붙이기 좋은 소재다.

## 데이터 한계를 정직하게 다룬 부분

이 프로젝트는 확보한 데이터가 실제로 무엇을 말해줄 수 있고 없는지를 숫자와 UI에
그대로 드러냈다. 데모에서 이 부분을 짚어주면 "그럴듯하게 포장하지 않았다"는 신뢰를
준다.

- **범위를 부산으로만 명확히 좁혔다.** 데이터 폴더에는 울산·경남 원본 파일도 있지만,
  법정동→행정동 매핑(좌표 기반 KDTree 매칭)을 검증까지 마친 건 부산뿐이라 그 이상은
  다루지 않았다. "일단 다 되는 것처럼 보이게" 하지 않고 검증된 범위만 서비스한 것.
- **남포동처럼 구 인구가 작은 동은 배후수요 추정이 특히 거칠다.** 인구 데이터가
  행정동이 아니라 구 단위로만 존재해서 모든 동에 소속 구 인구를 그대로 적용하는데,
  남포동이 속한 중구는 부산에서 인구가 가장 적은 구(3.6만 명, 다른 구의 1/10 수준)라
  상권은 활발해도 "배후인구"로는 낮게 잡힌다. 지도/팝업에 점선 테두리로 "구 단위
  추정치"라고 항상 표시한다.
- **접근성 지표는 데이터가 아예 없어 점수에서 뺐다.** 대중교통 정류장·집객시설
  근접도를 계산할 원본 데이터가 없다는 걸 확인하고, 억지로 대체 지표를 끼워 넣는
  대신 가중치를 나머지 지표에 재분배하고 UI에 "데이터 없음" 해치 패턴으로 표시했다.
- **Track A(ML 모델)는 실제 라벨로 학습했지만 성능이 약하다는 걸 그대로 보여준다.**
  일반음식점표준데이터의 실제 인허가일자/폐업일자로 "3년 내 폐업" 라벨을 만들어
  랜덤포레스트를 학습(AUC 0.656)했는데, 행정동 단위 집계 피처만으로는 개별 매장의
  운명을 정확히 맞추기 어렵고 Track B 가중합 점수와 순위 방향도 잘 안 맞는다
  (206개 행정동 전체 Spearman 상관 0.158, 약함). 이걸 숨기지 않고 상세 팝업의
  별도 접이식 "실험적 예측 모델" 섹션에 "신뢰도가 낮다"는 문구와 함께 분리해서
  보여준다 — 메인 점수·지도 색상에는 영향을 주지 않는다. 음식점(한식) 외 업종은
  애초에 폐업 이력 데이터가 없어 이 모델을 적용하지 않는다는 것도 명시한다.
- **LLM은 숫자를 다시 계산하지 않고 해설만 한다.** 프롬프트에서 점수 재계산을
  명시적으로 금지해, 리포트 문장과 실제 점수 로직이 항상 일치하게 만들었다.

### 예외 상황이 실제로 매끄럽게 처리되는지 점검한 것

- Gemini API 키를 일부러 무효화해서 테스트 → 실패를 감지해 점수 기반 기본 리포트로
  자동 전환되고, 화면에는 "AI 리포트 생성에 실패해 기본 요약을 표시합니다" 안내만
  뜬다(에러 화면이나 멈춤 없음)
- 백엔드를 완전히 내려서 테스트 → "서버에 연결할 수 없습니다" 같은 이해할 수 있는
  한국어 메시지로 안내(브라우저 원본 에러 문구 노출 안 함)
- 검색어가 아무 지역과도 안 맞을 때 → "~와 일치하는 지역이 없습니다" 안내
- 데이터가 없는 업종(카페/편의점/미용실)을 골라도 죽지 않고, 접근성처럼 "데이터
  없음"으로 투명하게 표시하며 나머지 점수로 계속 동작
- 데스크톱/태블릿/모바일 너비에서 콘솔 에러 없이 레이아웃이 깨지지 않는지 확인
  (모바일에서 상단바가 겹치던 문제와 지도가 눌리던 문제를 발견해 이번에 수정)

## 발표 스크립트

위 "데이터 한계" 섹션은 개발 문서로 그대로 두고, 여기는 발표 때 실제로 입으로
할 말을 정리한 것 — 강점은 자신 있게, 한계 질문은 짧고 담백하게 받아친다.

### 강조할 강점

- **"차려주는" 서비스다.** 점수판만 보여주고 끝나는 게 아니라, 자연어로 "서면에
  커피숍 차릴 건데 어디가 좋아?"라고 물으면 지역·업종을 알아서 파악해 바로
  분석까지 이어준다. 점수가 낮은 곳을 골라도 "여기 별로예요"에서 끝나지 않고,
  AI가 3km 이내에서 더 나은 곳을 직접 찾아 "왜 더 나은지" 근거를 대며 추천하고
  지도에서 눈으로 연결해 보여준다.
- **자연어 질의는 그냥 LLM한테 다 맡긴 게 아니다.** Gemini에 실제 206개 행정동
  이름 전체를 주고 "이 목록에 있는 이름만 골라라"라고 강제한 뒤, 응답도 실제
  목록과 다시 대조해서 검증한다. "서면"처럼 공식 행정동명이 아닌 생활권 이름도
  이해하면서, 동시에 없는 지명을 지어내지 못하게 막았다.
- **점수는 100% 결정론적이고, AI는 숫자를 손대지 않는다.** 배후수요·경쟁강도·
  수익성 가중합(Track B)이 실제 점수를 만들고, Gemini는 그 숫자를 절대 다시
  계산하지 않고 해설만 하도록 프롬프트에서 명시적으로 금지했다. 같은 지역·
  업종이면 언제 조회해도 같은 점수가 나온다.
- **진짜 정부 데이터로 실측 검증까지 했다.** Mock 데이터로 시작했지만, 좌표계
  변환(TM→WGS84)이 맞는지 실측으로 확인하고, 법정동→행정동 매핑을
  leave-one-out 교차검증으로 정확도를 확인한 뒤에야 실서비스에 반영했다.
  "일단 그럴듯하게 보이면 된다"가 아니라 매 단계 실제로 맞는지 검증했다.
  (부산 206개 행정동 전체, 실제 방문인구·소비매출·상가정보·폐업 데이터 기반)
- **AI 실패를 실제로 겪으면서 만든 서비스다.** 개발 중 Gemini 무료 티어 쿼터가
  실제로 소진되는 상황을 여러 번 겪었고, 그때마다 서비스가 멈추지 않고 점수
  기반 기본 리포트로 자동 전환되는 걸 직접 확인했다. 이론적 예외처리가 아니라
  실전에서 검증된 안정성이다.

### 예상 질문 & 답변

**Q. 왜 부울경 전체가 아니라 부산만 지원하나요?**
A. 경남·울산에는 부산과 동급으로 검증된 법정동→행정동 매핑 데이터가 없다는
걸 직접 조사로 확인했고, 데이터가 확실한 부산에만 집중했습니다. 확장은 같은
방식(좌표 기반 최근접 매칭)을 다른 지역에 적용하면 됩니다.

**Q. 이 점수, AI가 만든 거 아닌가요? 믿을 수 있나요?**
A. 아니요, 점수는 배후수요·경쟁강도·수익성을 가중합하는 결정론적 계산식입니다.
AI(Gemini)는 그 점수를 절대 다시 계산하지 않고 해설만 하도록 프롬프트에서
명시적으로 막아뒀습니다.

**Q. 접근성(교통) 점수는 왜 없나요?**
A. 대중교통 정류장이나 집객시설 위치 데이터가 확보한 자료 어디에도 없다는 걸
확인했습니다. 억지로 대체 지표를 넣는 대신, 나머지 지표에 가중치를 재분배하고
화면에 "데이터 없음"이라고 투명하게 표시했습니다.

**Q. 머신러닝(Track A) 모델은 왜 "신뢰도 낮음"이라고 써놨나요?**
A. 실제 폐업 데이터로 학습했지만, 행정동 단위 집계 데이터만으로는 개별 매장의
운명까지 정확히 맞추기 어렵다는 걸 검증 과정에서 확인했습니다. 이걸 숨기지
않고 "실험적 지표"로 분리해서 보여주는 게, 데이터를 있는 그대로 다룬다는
증거라고 생각합니다.

**Q. 인구 데이터가 구 단위라던데, 그럼 부정확한 거 아닌가요?**
A. 맞습니다. 그래서 배후인구가 구 단위 추정치인 지역은 지도와 팝업에 항상
점선 테두리로 표시해서, 사용자가 그 사실을 알고 판단하게 했습니다.

**Q. Gemini API가 느리거나 실패하면 어떻게 되나요?**
A. 타임아웃과 실패를 감지해 점수 기반 기본 템플릿으로 자동 전환됩니다. 실제로
개발 중 무료 티어 쿼터가 소진되는 상황을 여러 번 겪으면서 이 폴백 경로를
직접 검증했습니다.

**Q. 지금 바로 실사용 가능한 서비스인가요?**
A. MVP 단계입니다. 부산 206개 행정동, 4개 업종까지는 실데이터로 정확히
동작하고, 지역·업종 확장은 지금 쓰는 DataProvider 인터페이스 구조 그대로
데이터만 추가하면 됩니다.
